# dev-workflow プラグイン改善計画

## 概要

Claude Codeを用いた開発ワークフロー全体を体系化し、各フェーズで必要な作業をskillとして実装する。

---

## 1. 開発ワークフロー全体像

### 1.1 フェーズ構成

```
[理解] → ボリューム判定
            │
   ┌────────┼────────┐
   ↓        ↓        ↓
 タスク   ストーリー  エピック
   │        │        │
   │      spec作成  epic作成
   │        │        │
   │      plan作成  ストーリー分解
   │        │        │
   │        │        └→ [理解]に戻る（各ストーリー）
   │        │
   │   [セッションクリア]
   │        │
   └────────┴────────→ [実装]
                         ↓
                  [テスト/AIレビュー] ←┐
                         │           │イテレーション
                         └───────────┘
                         ↓
                  [ユーザーレビュー]
                         ↓
                     [コミット]
                         ↓
                   [ナレッジ化]
```

### 1.2 作業ボリュームによる分岐

| レベル | 判定基準 | 文書 | 実装開始方法 |
|-------|---------|------|-------------|
| タスク | 1セッションで完結 | なし | Claude Code Plan機能 |
| ストーリー | 複数セッションにまたがる | spec + plan | 文書から開始 |
| エピック | 複数ストーリーで構成 | epic + 各ストーリーのspec/plan | 各ストーリーごとに実行 |

### 1.3 文書の役割と更新頻度

| 文書 | 内容 | 更新頻度 |
|------|------|---------|
| epic | 要件整理 + ストーリー管理（実装状況含む） | 低 |
| spec | 要件 + 受け入れ基準（AIセルフレビュー用） | 中 |
| plan | 実装手順 + 進捗 | 高 |

---

## 2. 重要な設計方針

### 2.1 セッションクリア前提
- **実装フェーズ開始時は必ずセッションをクリア**する
- タスクの場合: Claude Code Plan機能が内部的に保持
- ストーリー/エピックの場合: spec + planが唯一の情報源

### 2.2 Self-contained文書
- すべての文書は会話履歴なしで自律実行可能な状態にする
- 現行ExecPlanの「self-contained」の考え方を継承

### 2.3 AIセルフレビュー可能な受け入れ基準
- specには検証可能な受け入れ基準を含める
- AIがセルフレビューで合否判定できる形式

### 2.4 再帰的フロー
- エピック → ストーリー分解後、各ストーリーは[理解]フェーズに戻る
- ボリューム判定を再度実施

### 2.5 失敗時の対応
- 失敗のナレッジを含めて文書を更新しつつ巻き戻す
- 各フェーズでその時点の結果を残す

## 3. Skill構成

### 3.1 プリタスク（理解〜計画）

| Skill | 役割 | トリガー例 |
|-------|------|-----------|
| `assess-task` | ボリューム判定、タスク/ストーリー/エピック分岐決定 | 「このタスクを始めたい」「作業量を判断して」 |
| `create-epic` | epic文書作成 + ストーリー分解支援 | 「エピックを作成して」「要件を整理して」 |
| `create-spec` | 要件 + 受け入れ基準の文書作成 | 「specを作成して」「仕様を書いて」 |
| `create-plan` | 実装手順の文書作成 | 「planを作成して」「実装計画を立てて」 |

### 3.2 実装中

| Skill | 役割 | トリガー例 |
|-------|------|-----------|
| `self-review` | specの受け入れ基準に基づくAIセルフレビュー | 「レビューして」「受け入れ基準をチェック」 |

### 3.3 ポストタスク

| Skill | 役割 | トリガー例 |
|-------|------|-----------|
| `post-task` | コミット + ナレッジ化（ナレッジベース/ADR）の統合処理 | 「タスク完了」「まとめて」 |

post-task内で状況に応じて提案:
- cleanup（デバッグコード削除、一時ファイル削除）

| `continuity-ledger` | plan文書の進捗管理でカバー、削除 |
| `workflow-orchestrator` (agent) | assess-taskに置き換え |

## 4. Skill詳細設計

### 4.1 assess-task

**目的**: タスクを理解し、ボリュームを判定して適切なフローに分岐

**トリガー**: 「このタスクを始めたい」「作業を開始する」「ボリュームを判断して」

**処理フロー**:
1. ユーザーからタスクの説明を受け取る
2. 明確でない点があれば質問で深掘り
3. 以下の基準でボリューム判定:
   - **タスク**: 1セッション完結、1-3ファイル、明確な要件、アーキテクチャ変更なし
   - **ストーリー**: 複数セッション、4+ファイル、または設計判断が必要
   - **エピック**: 複数の独立したストーリーに分解可能
4. 判定結果と理由を提示
5. 次のアクションを案内:
   - タスク → Claude Code Planモードへ
   - ストーリー → create-spec skillへ
   - エピック → create-epic skillへ

**出力**: 判定結果（タスク/ストーリー/エピック）+ 理由 + 次のアクション

---

### 4.2 create-epic

**目的**: epic文書を作成し、ストーリーに分解

**トリガー**: 「エピックを作成して」「要件を整理して」「大きな機能を分解して」

**処理フロー**:
1. 全体の要件・目的を整理
2. ストーリーに分解（各ストーリーは独立して実装可能な単位）
3. ストーリー間の依存関係を特定
4. 優先順位を設定
5. epic文書を作成（`docs/epic-{name}.md`）

**参照**: `references/epic-template.md`

**出力**: epic文書ファイル

---

### 4.3 create-spec

**目的**: 要件と受け入れ基準を文書化

**トリガー**: 「specを作成して」「仕様を書いて」「要件を定義して」

**処理フロー**:
1. 要件の詳細を整理
2. **AIセルフレビュー可能な受け入れ基準**を定義
   - 検証可能な形式（「〜が〜である」）
   - 具体的な入力と期待出力
   - エッジケースの明示
3. スコープ外を明示
4. spec文書を作成（`docs/spec-{name}.md`）

**参照**: `references/spec-template.md`

**出力**: spec文書ファイル

**重要**: specは実装セッションの唯一の情報源となるため、self-containedであること

---

### 4.4 create-plan

**目的**: 実装手順を文書化

**トリガー**: 「planを作成して」「実装計画を立てて」「どう実装するか整理して」

**処理フロー**:
1. specを読み込む
2. 実装のアプローチを決定
3. ファイル変更の一覧を作成
4. 実装ステップを順序立てて記述
5. plan文書を作成（`docs/plan-{name}.md`）

**参照**: `references/plan-template.md`

**出力**: plan文書ファイル

**重要**: planは頻繁に更新される。進捗セクションを含む

---

### 4.5 self-review

**目的**: specの受け入れ基準に基づいてAIセルフレビュー

**トリガー**: 「レビューして」「受け入れ基準をチェックして」「実装が完了したか確認」

**処理フロー**:
1. 対応するspec文書を読み込む
2. 受け入れ基準を1つずつ検証
3. 各基準について: PASS / FAIL / 要確認
4. FAILの場合は具体的な問題点と修正案を提示
5. 全基準PASSならユーザーレビューへ進むことを提案

**出力**: レビュー結果（各基準の合否 + 問題点 + 修正案）

---

### 4.6 post-task

**目的**: タスク完了後の処理を統合実行

**トリガー**: 「タスク完了」「まとめて」「コミットしてナレッジ化」

**処理フロー**:
1. **コミット準備**
   - 変更内容の確認
   - コミットメッセージ作成
2. **ナレッジ化**
   - 学んだこと、発見をナレッジベースに記録
   - 重要な技術的決定があればADR作成を提案
3. **追加アクション提案**（該当する場合）
   - cleanup: デバッグコード、一時ファイルの削除
   - update-docs: README、CHANGELOG更新
   - pr: PR作成

**参照**: 既存の`adr` skillのテンプレート部分

**出力**: コミット + ナレッジ記録 + 追加アクション

## 5. テンプレート詳細設計

### 5.1 epic-template.md

```markdown
# Epic: {タイトル}

## 概要
{このエピックで実現すること}

## 背景・動機
{なぜこのエピックが必要か}

## ストーリー一覧

| # | ストーリー | 状態 | 依存 |
|---|-----------|------|------|
| 1 | {ストーリー名} | 未着手/進行中/完了 | - |
| 2 | {ストーリー名} | 未着手 | #1 |

## スコープ外
{このエピックでは扱わないこと}

## 参考情報
{関連リンク、既存コードへの参照など}
```

### 5.2 spec-template.md

```markdown
# Spec: {タイトル}

## 概要
{何を実現するか}

## 背景
{なぜ必要か、コンテキスト}

## 要件

### 機能要件
- {要件1}
- {要件2}

### 非機能要件
- {パフォーマンス、セキュリティなど}

## 受け入れ基準

各基準はAIが検証可能な形式で記述:

1. **{基準名}**
   - 条件: {前提条件}
   - 操作: {実行する操作}
   - 期待結果: {検証可能な結果}

2. **{基準名}**
   - 条件: ...
   - 操作: ...
   - 期待結果: ...

## スコープ外
{このspecでは扱わないこと}

## 技術的な注記
{実装者が知っておくべき技術的な情報}
```

### 5.3 plan-template.md

```markdown
# Plan: {タイトル}

**関連spec**: `docs/spec-{name}.md`

## アプローチ
{実装の全体方針}

## 変更ファイル一覧

| ファイル | 変更内容 |
|---------|---------|
| `path/to/file.ts` | {変更内容} |

## 実装ステップ

### Step 1: {ステップ名}
{具体的な作業内容}

### Step 2: {ステップ名}
{具体的な作業内容}

## 進捗

- [x] Step 1 完了
- [ ] Step 2
- [ ] Step 3

## 決定ログ
{実装中に下した決定とその理由}

- {日付}: {決定内容} - {理由}

## 発見・学び
{実装中に発見したこと}
```

## 8. 検証方法

1. 各skillが適切なトリガーで発動するか確認
2. タスク/ストーリー/エピックそれぞれのフローを実際に試行
3. セッションクリア後に文書から実装開始できるか確認
4. AIセルフレビューが受け入れ基準に基づいて動作するか確認
